## Why this happened (best hypothesis)

The Edit tool replaces an old_string with a new_string by writing the entire file. On a 2800-line file with a long new_string containing many backtick template literals and unicode characters (em-dashes, smart quotes), the write may not have flushed the full buffer before the tool returned success. The truncation point being mid-word ("SourcingRe…", "b…") supports this — it's a byte-buffer cut, not a logical edit boundary.

**Mitigation for next round:**
- Smaller Edits — change a function at a time, not multi-block diffs
- Verify line count after every Edit via `wc -l`
- Run `tsc --noEmit` after every batch (not just at end-of-batch)

This is now documented in the orchestrator's hard rules for the daily scheduled task — see `C:\Users\Iceberg\Claude\Scheduled\lab2date-recovery-orchestrator\SKILL.md`.
