## Code changes

### 1. `src/app/admin/actions.ts` — BUG-022 fix (P1, Financial/State-integrity · invariant F12)
**Why:** `setOrderFulfillment` already guards address-completeness (BUG-009),
idempotency + no-op fast path (BUG-010), and concurrent races via
`updateMany({ where: { id, status: order.status } })`. But that atomic precondition
only asserts the status hasn't changed concurrently — it does **not** prevent a
transition *out of* a terminal money-state. An order already **REFUNDED** or
**CANCELED** could be flipped to **PROCESSING / SHIPPED / DELIVERED** (the `where`
clause matches `status === order.status === 'REFUNDED'`), shipping goods against
money already returned and emailing the buyer that their refunded order shipped.
This is invariant **F12** (was ⏳) and part of **S3**.

**Change:** Added an early guard immediately after the order fetch, before the
tracking-number auto-bump and the atomic transition:

```ts
const TERMINAL_ORDER_STATES = new Set(['REFUNDED', 'CANCELED']);
if (TERMINAL_ORDER_STATES.has(order.status) && status !== order.status) {
  throw new Error(
    `Cannot change order ${order.orderNumber} to ${status.toLowerCase()} — it is ` +
    `${order.status.toLowerCase()} (terminal). Refunded or canceled orders cannot ` +
    `be re-fulfilled.`,
  );
}
```

- Only a status **change** out of REFUNDED/CANCELED is blocked. Same-status edits
  (e.g. attaching a tracking note to a refunded row) still pass.
- Refund/cancel keep their dedicated actions (`refundOrder`, `cancelOrder`), which
  own those transitions and the stock restock — untouched.
- Scope kept tight: no broader S3 forward-only-monotonicity rewrite (no drive-by
  refactor). DELIVERED→backwards and full monotonicity remain a follow-up.
- Manual-payment-safe: no Stripe/card/pay-now wording added.
