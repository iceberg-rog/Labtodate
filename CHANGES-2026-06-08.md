## Files changed

### 1. `src/app/admin/actions.ts` — BUG-026 (P1, state-integrity / invariant S3)
- `setOrderFulfillment`: added a **forward-only monotonicity guard**, placed
  after the tracking auto-bump and before the side-effecting `updateMany`.
  It defines the fulfilment funnel `['PAID','PROCESSING','SHIPPED','DELIVERED']`
  and, **only when the status actually changes**, rejects:
  1. `CANCELED`/`REFUNDED` targets — bounced to the dedicated Refund/Cancel
     actions so reserved stock is restocked correctly;
  2. fulfilment of a not-yet-paid order (current status outside the funnel,
     e.g. `PENDING_PAYMENT`) — operator must record payment first
     (`markOrderPaidManually`/`verifyPayment`), which also avoids creating a
     `PAID` row with `paidAt=null` (F14);
  3. any backward move (`rank(target) < rank(current)`), e.g. DELIVERED→PROCESSING,
     SHIPPED→PAID.
- Same-status edits (attach/adjust tracking on an existing row) remain allowed —
  the existing BUG-010 idempotency no-op path is untouched.
- The pre-existing BUG-009 (address completeness), BUG-010 (idempotency + atomic
  transition), and BUG-022 (terminal-exit) guards are all preserved.
- No change to `markOrderPaidManually`, `verifyPayment`, `refundOrder`,
  `cancelOrder`, or `bulkMarkAllShipped` (the latter's WHERE already only moves
  `PAID/PROCESSING → SHIPPED`, which is forward-safe). Scope kept tight — no
  drive-by refactor.
